# Visitor Pattern

Mi Compilador uses the **visitor pattern** to traverse the Abstract Syntax Tree (AST) and execute instructions. This pattern separates tree traversal logic from execution logic, making the code clean and extensible.

## What is the Visitor Pattern?

The visitor pattern allows you to:
- **Traverse** the AST nodes
- **Execute** operations on each node type
- **Return values** from node visits
- **Separate concerns**: parsing vs. execution

## The Visitor Class Hierarchy

```text
ExpresionesVisitor (Base Class - Generated by ANTLR)
    ├── visitProg(ctx)
    ├── visitInstrAsig(ctx)
    ├── visitAritmetica(ctx)
    └── ... (other visit methods)

Visitor (Custom Implementation - Extends ExpresionesVisitor)
    ├── memoria: dict (runtime variable storage)
    ├── visitProg(ctx)
    ├── visitInstrAsig(ctx)
    ├── visitAritmetica(ctx)
    ├── visitNumero(ctx)
    ├── visitVariable(ctx)
    └── ... (all overridden visit methods)
```

### Base Visitor Class

ANTLR generates `ExpresionesVisitor.py` with stub methods:

```python
class ExpresionesVisitor(ParseTreeVisitor):
    def visitProg(self, ctx):
        return self.visitChildren(ctx)
    
    def visitInstrAsig(self, ctx):
        return self.visitChildren(ctx)
    
    # ... more visit methods
```

### Custom Visitor Implementation

We override methods in `Visitor.py` to implement execution logic:

```python Visitor.py lines
from ExpresionesVisitor import ExpresionesVisitor
from ExpresionesParser import ExpresionesParser

class Visitor(ExpresionesVisitor):
    def __init__(self):
        self.memoria = {}  # Runtime memory for variables
```

## How Visitor Methods Work

### The Context Object

Each `visit` method receives a **context object** (`ctx`) that represents a parsed grammar rule:

```python
def visitInstrAsig(self, ctx):
    # ctx is an InstrAsigContext object
    # It provides access to child nodes and tokens
```

### Context Methods

| Method | Purpose | Example |
|--------|---------|----------|
| `ctx.ID()` | Get identifier token | `ctx.ID().getText()` → `"x"` |
| `ctx.expr()` | Get expression node | `self.visit(ctx.expr())` |
| `ctx.getChild(n)` | Get nth child | `ctx.getChild(1).getText()` → `"+"` |
| `ctx.getText()` | Get all text | `ctx.getText()` → `"x=10"` |

## Visit Method Implementations

### visitProg - Program Entry Point

```python Visitor.py
def visitProg(self, ctx):
    return self.visitChildren(ctx)
```

- Entry point for the entire program
- Visits all instruction children
- Returns the result of the last instruction

### visitInstrAsig - Variable Assignment

```python Visitor.py
def visitInstrAsig(self, ctx):
    nombre_var = ctx.asignacion().ID().getText()
    valor = self.visit(ctx.asignacion().expr())
    self.memoria[nombre_var] = valor
    print(f"Asignación: {nombre_var} = {valor}")
    return valor
```

<Steps>
  <Step title="Extract variable name">
    Get the identifier from the assignment context
  </Step>
  <Step title="Evaluate expression">
    Recursively visit the right-hand side expression
  </Step>
  <Step title="Store in memory">
    Save the value in the memory dictionary
  </Step>
  <Step title="Print output">
    Display the assignment for debugging
  </Step>
</Steps>

### visitAritmetica - Arithmetic Operations

```python Visitor.py
def visitAritmetica(self, ctx):
    izq = self.visit(ctx.expr(0))  # Left operand
    der = self.visit(ctx.expr(1))  # Right operand
    op = ctx.getChild(1).getText()  # Operator
    
    # Safety check for None values
    if izq is None or der is None:
        return 0
    
    if op == '+': return izq + der
    if op == '-': return izq - der
    if op == '*': return izq * der
    if op == '/': return izq // der  # Integer division
    return 0
```

**Key points:**
- `ctx.expr(0)` - Left expression
- `ctx.expr(1)` - Right expression  
- `ctx.getChild(1)` - The operator token (middle child)
- Recursively evaluates both sides
- Returns the computed result

<Note>
The division operator uses `//` for integer division since the language only supports integers.
</Note>

### visitNumero - Number Literals

```python Visitor.py
def visitNumero(self, ctx):
    return int(ctx.NUM().getText())
```

- Extracts the number token text
- Converts to Python integer
- Returns the numeric value

### visitVariable - Variable Lookup

```python Visitor.py
def visitVariable(self, ctx):
    nombre = ctx.ID().getText()
    if nombre in self.memoria:
        return self.memoria[nombre]
    print(f"Error: Variable '{nombre}' no definida.")
    return 0
```

- Retrieves variable value from memory
- Returns the stored value
- Prints error if undefined (returns 0 as fallback)

### visitParentesis - Parentheses

```python Visitor.py
def visitParentesis(self, ctx):
    return self.visit(ctx.expr())
```

- Unwraps parentheses
- Evaluates the inner expression
- Returns the result unchanged

### visitInstrIf - Conditional Statements

```python Visitor.py
def visitInstrIf(self, ctx):
    condicion = self.visit(ctx.exprRelacional())
    if condicion:
        return self.visit(ctx.bloque(0))  # Then block
    elif ctx.bloque(1):
        return self.visit(ctx.bloque(1))  # Else block
    return None
```

- Evaluates the condition
- Visits the `then` block if true
- Visits the `else` block if false (if present)
- Returns the result of the executed block

### visitRelacional - Relational Expressions

```python Visitor.py
def visitRelacional(self, ctx):
    izq = self.visit(ctx.expr(0))
    der = self.visit(ctx.expr(1))
    op = ctx.op.text  # The operator captured in grammar
    
    if op == '>': return izq > der
    if op == '<': return izq < der
    if op == '==': return izq == der
    if op == '!=': return izq != der
    if op == '>=': return izq >= der
    if op == '<=': return izq <= der
    return False
```

- Evaluates both sides
- Compares using the specified operator
- Returns a boolean result (`True`/`False`)

<Note>
The grammar uses `op=( '>' | '<' | ... )` to capture the operator, making it accessible via `ctx.op.text`.
</Note>

### visitBloque - Code Blocks

```python Visitor.py
def visitBloque(self, ctx):
    return self.visitChildren(ctx)
```

- Visits all instructions inside the block
- Returns the result of the last instruction
- Used for if/else bodies

## Visitor Execution Flow

For this program:
```text
program {
    int x;
    x = 5 + 3;
}
```

Execution trace:
1. `visitProg()` - Start execution
2. `visitInstrAsig()` - Process assignment
3. `visitAritmetica()` - Evaluate `5 + 3`
4. `visitNumero()` - Return `5`
5. `visitNumero()` - Return `3`
6. `visitAritmetica()` - Return `8` (5 + 3)
7. `visitInstrAsig()` - Store `x = 8`

## Adding New Visitor Methods

To add support for a new language feature:

<Steps>
  <Step title="Update the grammar">
    Add a new rule with a label in `Expresiones.g`
  </Step>
  <Step title="Regenerate the parser">
    Run `antlr4` to regenerate visitor classes
  </Step>
  <Step title="Implement the visit method">
    Add the new method to `Visitor.py`
  </Step>
  <Step title="Test the feature">
    Create test programs to verify behavior
  </Step>
</Steps>

Example - Adding a print statement:

```python
# 1. Grammar rule (with #InstrPrint label)
instrucciones
    : 'print' '(' expr ')' ';'  #InstrPrint

# 2. Visitor method
def visitInstrPrint(self, ctx):
    valor = self.visit(ctx.expr())
    print(f"Output: {valor}")
    return valor
```

## Visitor Pattern Best Practices

<Tip>
**Always check for None values** when visiting expressions. Some operations may return `None` for undefined variables or errors.
</Tip>

<Tip>
**Return meaningful values** from visitor methods. These values propagate up the tree and can be used by parent nodes.
</Tip>

<Warning>
**Don't modify the AST** during traversal. The visitor pattern assumes the tree is immutable.
</Warning>

## Key Takeaways

- The visitor pattern separates tree traversal from execution logic
- Each grammar label generates a corresponding `visit` method
- Context objects provide access to child nodes and tokens
- Visitor methods can call other visitor methods recursively
- The pattern makes it easy to add new features
